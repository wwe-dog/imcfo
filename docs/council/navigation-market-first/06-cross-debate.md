# Cross-Debate

## 1. Product ↔ User Behavior

Question: Which model has Personal CFO identity while staying understandable?

Position A:

- Product Doctrine favors a model that makes "经营自己" the first mental model.
- Ordinary bookkeeping navigation would erase differentiation.

Challenge B:

- User Behavior warns that "经营" alone can be abstract.
- Users need concrete first questions: safe, can spend, what changed, what next.

Resolution:

- Use "经营" as the main destination only if it contains concrete CFO brief content.
- First-run and empty states should explain it through user questions, not theory.

Rule / Model Impact:

- Operating CFO Loop remains strongest, but must be tested against "今日 CFO" and "今日判断".

## 2. Product ↔ Originality Judge

Question: Which model is least generic and most communicable?

Position A:

- Product Doctrine argues that "说一笔 -> 看影响 -> 看经营" is the most unique loop.

Challenge B:

- Originality Judge rejects any model that merely renames ledger tabs.
- It also rejects pure chat because many AI apps already use chat as wrapper.

Resolution:

- Differentiation comes from a loop: life event translation, operating state, trusted report, action.

Rule / Model Impact:

- Top-level should avoid "交易 / 预算 / 账户 / 报表 / 我的" as identity.

## 3. AI Input ↔ User Behavior

Question: Should AI input be center, entry, action, or invisible capability?

Position A:

- AI Input Agent says AI is center capability and global action, not a normal page.

Challenge B:

- User Behavior says if it is not visible enough, users will not discover the low-friction path.

Resolution:

- Use a persistent global primary action "说一笔".
- Add contextual AI affordances inside 经营 / 报告 / 账本, but do not create AI as the only destination.

Rule / Model Impact:

- "说一笔" should be globally reachable, with draft recovery and manual fallback.

## 4. AI Input ↔ Financial Core

Question: Where does feedback after AI input belong?

Position A:

- AI Input Agent says drafts can contain estimate preview, but not final report output.

Challenge B:

- Financial Core requires posted facts and report engine authority.

Resolution:

- Before confirmation: show AI estimate as draft impact.
- After posting: show rule-layer / report-engine result.
- If they differ, report engine wins.

Rule / Model Impact:

- Navigation must separate Draft, Posted Fact, Report Result and CFO Explanation.

## 5. Financial Core ↔ Product

Question: Should three statements be explicit navigation?

Position A:

- Financial Core wants formal reports accessible and authoritative.

Challenge B:

- Product Doctrine warns that "资产负债表 / 利润表 / 现金流量表" as top-level tabs may intimidate ordinary users.

Resolution:

- Use "报告" as top-level trust layer.
- Inside it support Simple Mode and Professional Mode.
- Three statements remain product skeleton, not necessarily first-level labels.

Rule / Model Impact:

- Financial Statement Model becomes trust layer, not primary navigation.

## 6. Architecture ↔ Overengineering Critic

Question: Which model is unique but not overbuilt?

Position A:

- Architecture Contract Agent prefers stable destinations plus global action: 经营 / 报告 / 账本 / 我的 + 说一笔.

Challenge B:

- Overengineering Critic warns that Operating System and Coach models expand too quickly.

Resolution:

- Four destinations plus one global action is the current strategic recommendation.
- Keep future modules behind experiments.

Rule / Model Impact:

- No enterprise-style menu; no universal task system in MVP.

## 7. Scope Judge ↔ Innovation Freedom

Question: What is current validation and what stays future?

Position A:

- Scope Judge allows market-first strategy but rejects locking a full operating system.

Challenge B:

- Innovation Freedom requires no navigation model to become permanent constitution.

Resolution:

- Constitution may receive principles only: navigation should serve the financial operating loop, preserve traceability, and keep AI drafts separate.
- Concrete labels and tab structure belong to experiments or future specs after user testing.

Rule / Model Impact:

- This report is a recommendation, not a binding spec.
