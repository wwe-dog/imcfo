# IMCFO V0.1 输入系统目标：智能记一笔

本文档记录 V0.1 输入窗口的产品目标：语音 / 自然语言 / 记一笔入口。它是当前交接包中“V0.1 默认不接真实 AI/API”边界的有限调整，只允许真实 API 服务于“生活语言 → 结构化候选交易 Draft”这一段。

## 1. 产品级定位

IMCFO V0.1 的核心卖点从普通“记账入口”升级为“智能记一笔”。

记一笔不是管理页中的附属功能，而是 IMCFO 从用户生活事件进入个人公司财务系统的核心入口。

核心表达：

用户可以用语音或自然语言描述一件生活事件，系统将其识别为一笔结构化候选交易，并在用户确认后，按照个人公司视角更新资产、负债、收入、费用、利润和现金流。

一句话定义：

> 说一句生活话，IMCFO 把它翻译成一笔个人公司的财务记录。

## 2. V0.1 核心链路

标准链路必须是：

```text
语音输入 / 文本输入
→ 识别中状态
→ 结构化候选交易 Draft
→ 用户确认或修改
→ 调用现有 useAppData / transactionRules
→ 正式入账
→ 首页、报表、交易记录同步刷新
```

严禁做成：

```text
用户输入
→ API 或 UI 直接入账
```

API 或识别逻辑只能生成候选交易，不能直接写账。

## 3. 候选交易 Draft 必须包含的字段

识别结果确认卡至少包含：

- 金额
- 收支方向
- 分类
- 账户
- 日期
- 备注
- 置信度
- 会计影响摘要

示例：

用户输入：

```text
今天午餐 32 元，用支付宝
```

候选结果：

- 金额：¥32
- 方向：支出
- 分类：餐饮
- 账户：支付宝
- 日期：今天
- 会计影响：费用增加，现金减少，经营活动现金流出

## 4. 真实 API 边界调整

原交接包中 V0.1 默认不接真实 AI/API。现在产品方向发生边界调整：

V0.1 允许只为“记一笔识别链路”有限接入真实 API。

允许范围：

- 语音转文字 API
- 自然语言交易识别 API
- 将用户输入解析为结构化候选交易
- 返回金额、方向、账户、分类、日期、备注、置信度

禁止范围：

- 泛 AI 聊天
- AI 财务顾问
- 自动财务分析
- 自动投资建议
- 自动税务申报
- 发票识别
- 支付能力
- 登录体系
- 云同步账本
- 数据库后端
- 会员系统
- 个体工商户、企业主体、家庭合并账本逻辑

核心原则：

真实 API 只服务于“生活语言 → 结构化候选交易”这一段，不扩展为泛 AI 财务系统。

## 5. 架构边界

后续实现时必须遵守：

- Screen 不得直接调用 AsyncStorage。
- Screen 不得直接写账本。
- UI 不得发明会计公式。
- API 不得直接生成最终会计分录。
- API 不得直接修改资产、负债、收入、费用、利润、现金流。
- 用户确认后，正式入账必须继续走现有 useAppData 和 transactionRules。
- 会计公式、现金流规则、storage schema、route keys、domain model fields 不得随意修改。

推荐未来模块方向：

`recordRecognitionService`

负责：

- 接收语音转文字结果或文本输入。
- 调用本地规则或真实 API。
- 返回候选交易 Draft。
- 附带置信度和待确认字段。

该模块只负责识别，不负责入账。

## 6. V0.1 产品优先级

P0：

- 语音 / 文本记一笔主入口
- 识别中状态
- 识别结果确认卡
- 用户可修改金额、分类、账户、日期、备注
- 确认后入账
- 入账成功后展示本笔对资产、收入、费用、现金流的影响

P1：

- 首页增加快速记一笔入口
- 管理页强化智能记一笔入口
- 低置信度识别时引导用户选择
- 常见语句本地规则兜底

P2：

- 截图识别
- 更复杂的多笔交易识别
- 历史偏好学习
- 真实 AI 能力增强

## 7. 产品文案原则

用户可见文案必须中文。

不要把功能写成“AI 记账助手”或“财务顾问”。

推荐表达：

- 智能记一笔
- 说一句，自动识别
- 确认后入账
- 本笔影响
- 费用增加
- 现金减少
- 经营活动现金流出
- 收入增加
- 资产增加

避免表达：

- 自动帮你理财
- AI 财务建议
- 自动生成财务规划
- 自动税务处理
- 自动投资建议
