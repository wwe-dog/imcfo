# Specs

Specs describe current implementation and product behavior.

Boundary with discovery: discovery records what currently exists in the codebase. Specs record what the product should present and how it should behave. The two must not duplicate each other. If something is only in code and has no defined product behavior, it belongs in discovery. If something defines intended behavior regardless of current implementation state, it belongs in specs.

Specs may change as the product changes. They are not permanent constitution.

Use specs for:

- current page structure
- current interaction behavior
- current copy and information hierarchy
- current visual implementation details
- current supported transaction forms
- current API endpoint behavior
- current QA checklists

Do not use specs to override Constitution v2 core invariants.
